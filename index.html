BDD: a refresher

BDD = Behaviour Driven Development

Why should we care about using BDD?

...because we're moving towards being a Product focused company that delivers what customers need. BDD forces an outside in development process that requires you to think of the customer first, build the right software and close gaps in understanding

In Test Driven Development (TDD), a developer writes a failing test - typically at a Unit or Integration test level. They then write the minimum viable code to make the test pass. The rationale is that lean software is developed that meets requirements

In BDD = TDD extended. PO's, BA's, Testers and Developers collaborate to describe the requirements as 'Behaviours'. These behaviours are written as Scenarios in a language called Gherkin:

Feature: My new Feature
Given: I have an account setup
When: I have try to login
Then: I see the 'Welcome' page

Scenarios and Gherkin are, deliberately, non-technical languages that all business users can understand. The Testers *AND* Developers then convert Scenarios directly into tests, most commonly using the tool Cucumber.

  • the test describes a behaviour at a level the business understands
  • the test typically describes technology at the top of the stack - enforcing outside in development
  • the test must have code written to make it pass
  • the behaviours agreed by the team therefore translate into software that meets the stakeholders desires - build the right thing

The legacy of using BDD with Cucumber


  • living documentation - an 'executable specification'

  • a legacy of acceptance tests to act as a regression suite for the completed software

  • a set of acceptance tests that everyone understands: scenarios

  • enforces a 'Ubiquitous Language': the business and technology people end up talking a similar language, closing gaps in understanding and mitigating risks of missing requirements

    What is Cucumber?

    The mission statement from the developers of Cucumber:

    To provide a language, a process, and a tool that would provide a single source of truth of software behavior for the audience of both non-technical and technical project members

    • put simply - an open source tool to help create automated acceptance tests

    • no dashboard or gui - simply a bunch of jars

    • we call them acceptance tests because they express what the software needs to do in order for the stakeholder to find it acceptable.

    • the tests are written collaboratively between test, development and stakeholders: 3 Amigos

    • the test fails at the time of writing, because no code has been written yet, but it captures what the stakeholder cares about and gives everyone a clear signal as to what it will take to be done.

    • unit tests ensure you build the thing right, whereas acceptance tests ensure you build the right thing.

    • Cucumber tests interact directly with the developers’ code, but they’re written in a medium and language that business stakeholders can understand.

    In Cucumber

    • Test Suite = Feature
    • Test Cases = Scenarios
    • Test Steps = Steps
    • Test Code = Step Definitions + support code

    When you run Cucumber, it reads in your specifications from plain-language text files called features, examines them for scenarios to test, and runs the scenarios against your system.

    Why are we using Cucumber in the Test Team?

    Problem:

    • large number of test cases to run per supplier
    • had to pass in lots of test data - how?
    • tests were opaque; not easy to see what they were testing
    • lots of duplication & repetition

    Solution:

    • cucumber lets us write tests which do what they say on the tin
    • same Scenarios can be run against many suppliers
    • Scenarios can use Data Tables to run many times with different data inputs (reducing duplication
    • Scenarios can use parameters to pass in data at runtime
    • Test Team familiarises with terminology and tooling used in BDD
    • Cucumber reporting is clear and can easily highlight failures
    • We can tag tests using bespoke tags such as @Smoke, @Regression, @Fast...and run suite accordingly
